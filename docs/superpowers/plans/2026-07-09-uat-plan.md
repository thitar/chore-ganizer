# UAT: Automated + Manual Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Get the existing `e2e/` Playwright suite back to a known-good state, extend it to cover the M1 (dark redesign) and M2 (gamification/CSRF) work that shipped since the suite was last touched, and produce a manual click-through checklist for the user.

**Architecture:** Audit-first — run what exists before writing anything new, since the suite predates the entire v3.2.0 milestone and may already be broken against current selectors/routes. New specs follow the existing file-per-phase-or-flow convention. The manual checklist is a separate markdown deliverable, not test code.

**Tech Stack:** Playwright (already configured, `playwright.config.ts`), against a real running instance (dev servers or docker-compose), using the pattern established this session: an isolated throwaway SQLite DB seeded fresh, never the production data.

## Global Constraints

- **Never run e2e tests against production data.** Every test run in this plan uses an isolated DB (either the existing `test-db/integration-test.db` pattern from backend integration tests, or a fresh throwaway DB per the setup in Task 1 Step 1).
- **The backend must run with `NODE_ENV` set to anything other than `test` (e.g. `development`) for every Task 4 run.** `backend/src/middleware/csrf.ts` short-circuits entirely under `NODE_ENV=test` — running Task 4's CSRF spec with `NODE_ENV=test` set (accidentally inherited from the shell, a wrapping script, or CI) makes it pass without exercising CSRF at all, silently defeating the point of the spec. Confirm with `echo $NODE_ENV` before every Task 4 run; it must print `development` (or be unset with the backend's own default), never `test`. Note this is not a concern for Tasks 2-3 (M1/existing suite) since they don't depend on CSRF enforcement.
- Work on branch `test/uat-suite-refresh`.
- `.spec.ts` suffix for e2e tests, per `AGENTS.md`'s existing convention.
- One commit per spec file fixed or added.

## Task 1: Audit the existing suite

**Files:** none modified — this task produces a findings list consumed by Tasks 2-3.

- [ ] **Step 1: Get a running instance with a seeded DB**

Set up an isolated backend + frontend instance, never touching the production docker containers:

```bash
SCRATCH=/tmp/uat-suite-refresh  # or any scratch dir outside the repo
mkdir -p "$SCRATCH"

cd backend
export DATABASE_URL="file:$SCRATCH/uat-test.db"
export PORT=3011
export NODE_ENV=development
export SESSION_SECRET="uat-verification-only-secret-not-for-prod"
export SECURE_COOKIES=false
npx prisma db push --skip-generate
npx prisma db seed
nohup npm run dev > "$SCRATCH/backend-dev.log" 2>&1 &
```

Then temporarily point the frontend dev server's proxy at port 3011 (edit `frontend/vite.config.ts`'s `server.proxy['/api'].target` from `http://localhost:3010` to `http://localhost:3011` — revert this edit before committing anything in this plan), and start it:

```bash
cd ../frontend
nohup npm run dev > "$SCRATCH/frontend-dev.log" 2>&1 &
```

Playwright's `baseURL` (see `playwright.config.ts`) should already point at the frontend dev server's port; confirm before running tests. Seeded login: `dad@home.local` / `password123` (parent), `alice@home.local` / `password123` (child) — verify these are still accurate against `backend/prisma/seed.ts` before relying on them.

- [ ] **Step 2: Run the full existing suite against it**

```bash
npx playwright test --reporter=list
```

Record pass/fail per spec file: `path-a-regression.spec.ts`, `phase-04-uat.spec.ts`, `phase-05-points-happy-path.spec.ts`, `phase-05-uat.spec.ts`, `phase-06-uat.spec.ts`, `phase-07-uat.spec.ts`, `phase-10-uat.spec.ts`.

- [ ] **Step 3: For each failure, classify the cause**

Read the failure output and the spec file. Classify each as one of:
- **Stale selector** — targets an element/text that changed in the M1 dark redesign (e.g. old class names, old copy).
- **Stale route** — targets a URL or flow that moved.
- **Genuine regression** — the app actually behaves differently than the spec expects, and the spec is still correct about what SHOULD happen.
- **Obsolete** — the spec tests a flow that no longer exists (e.g. an old admin dashboard route from pre-rewrite).

Write this classification inline as a comment at the top of each failing spec file, so Task 2 doesn't have to redo the investigation.

- [ ] **Step 4: Check for coverage gaps**

Cross-reference the spec file names (all named after pre-rewrite phase numbers 4-10, plus one "path-a-regression") against what actually exists today. Confirm there is **no coverage at all** for:
- M1 "The Look": dark theme rendering, TopNav/BottomTabBar navigation, leaderboard
- M2 "The Game": streak display, level bar, badge grid, badge-earned celebration, CSRF-protected mutations (login, complete chore, adjust points)
- The CSRF fix itself (PR #146/#147) — there is currently no e2e test proving a real browser session can complete a chore end-to-end; only backend unit tests exist for the CSRF middleware logic itself.

## Task 5: Produce the manual UAT checklist

**Files:**
- Create: `docs/UAT-CHECKLIST.md`

- [ ] **Step 1: Draft the checklist structure**

One section per user-facing flow, each item phrased as a concrete action + expected result (not "test chores work" but "click Mark Complete on a pending chore → row shows Completed, points balance increases by the chore's point value"). Cover:

1. **Auth** — login as each seeded role (parent, child), logout, session persists across refresh, child redirected away from parent-only routes.
2. **Chores (child)** — view assigned chores, mark complete, see points increase.
3. **Chores (parent)** — create a chore template, assign a one-off chore, edit an assignment, delete an assignment, complete/uncomplete on behalf of a child.
4. **Recurring chores** — create a recurring chore, verify occurrences generate on the calendar/chores list, complete an occurrence.
5. **Points** — view balance and history, parent adjusts points (positive and negative), leaderboard updates.
6. **Gamification** — complete enough chores to earn a badge, verify the celebration toast + confetti fires, verify streak updates weekly, verify level bar progresses.
7. **Notifications** (requires `NTFY_BASE_URL` configured and a subscribed ntfy topic — skip this section entirely if ntfy isn't set up in the environment you're testing) — set an ntfy topic on profile, verify a chore-assigned push arrives, verify a badge-earned push arrives (real ntfy server, per the pattern used in PR #146's verification).
8. **Profile/family management** (parent) — view/edit family members, set a family member's ntfy topic, handle the "topic already in use" conflict.
9. **Mobile viewport** — resize to ~390px width, spot-check Dashboard/Points/Profile for layout breakage (this was manually verified once already for M2; the checklist should make it a repeatable step, not a one-off).
10. **Calendar** — view a month, navigate between months, verify assignments render on the correct day with correct colors.

- [ ] **Step 2: Write the file**

Each item as a markdown checkbox (`- [ ] `) so the user can literally check them off while clicking through.

- [ ] **Step 3: Commit**

```bash
git add docs/UAT-CHECKLIST.md
git commit -m "docs: add manual UAT checklist for click-through verification"
```
