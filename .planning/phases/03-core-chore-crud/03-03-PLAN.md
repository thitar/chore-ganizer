    **Service: complete(assignmentId, userId)**
    - Input: `assignmentId: number, userId: number` (the user performing the action, from session)
    - Behavior (CRITICAL — RESEARCH Pitfall 4: must be transactional with re-fetch):
      1. Find assignment with `include: { template: true }`. Throw 404 if not found.
      2. Check `assignment.assignedToId !== userId` → throw `AppError('You can only complete your own assignments', 403)` (ASVS V4).
      3. Check `assignment.status === 'COMPLETED'` → throw `AppError('Assignment is already completed', 409)`.
      4. Snapshot: `const pointsAwarded = assignment.template.points` (D-02).
      5. Use callback form of `prisma.$transaction` to atomically update + create PointLog, then re-fetch with includes:
         ```typescript
         const updated = await prisma.$transaction(async (tx) => {
           await tx.choreAssignment.update({
             where: { id: assignmentId },
             data: { status: 'COMPLETED', completedAt: new Date(), pointsAwarded },
           })
           await tx.pointLog.create({
             data: { userId: assignment.assignedToId, amount: pointsAwarded, type: 'EARNED', reason: assignment.template.title },
           })
           return tx.choreAssignment.findUnique({
             where: { id: assignmentId },
             include: { template: { select: { id: true, title: true, points: true, category: true } }, assignedTo: { select: { id: true, name: true, color: true } } },
           })
         })
         return updated
         ```
         The callback form ensures the returned assignment has `template` and `assignedTo` includes — without these, the frontend would crash on `assignment.template.title`. The array form `$transaction([...])` returns raw Prisma results with no `include`s.
      6. Return the fully-included updated assignment.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API (POST/PUT/DELETE /api/assignments) | Only PARENT role may create/edit/delete; Zod validates bodies |
| client→API (POST /api/assignments/:id/complete) | Any authenticated user, but service enforces ownership (assignedToId === userId) |
| client→API (POST /api/assignments/:id/uncomplete) | PARENT only; reverses PointLog |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-02 | Elevation of Privilege | Child creating/deleting/editing assignments via API | mitigate | `authorize('PARENT')` on POST/PUT/DELETE + uncomplete routes; CHILD gets 403 (D-18) |
| T-3-03 | Elevation of Privilege | Child marking another user's assignment complete | mitigate | Service checks `assignment.assignedToId === userId` before completing; returns 403 if mismatch (ASVS V4) |
| T-3-04 | Tampering | Non-atomic complete (PointLog not created if update succeeds but create fails) | mitigate | `prisma.$transaction([update, create])` wraps both operations; partial failure rolls back both (RESEARCH Pitfall 4, D-07) |
| T-3-05 | Tampering | Mass assignment via req.body on create/update | mitigate | Zod `createAssignmentSchema`/`updateAssignmentSchema` define exact allowed fields; validate() middleware strips unknowns |
| T-3-06 | Tampering | SQL injection via free-text category or template title in PointLog reason | mitigate | Prisma parameterized queries; no raw SQL anywhere. PointLog `reason` is set from `template.title` via Prisma create data object |
| T-3-07 | Tampering | Deleting a completed assignment by skipping status check | mitigate | Service checks `assignment.status === 'COMPLETED'` before delete; returns 409 if true (D-13) |
| T-3-08 | Information Disclosure | User list reveals emails of family members | mitigate | `GET /api/users` uses `select: { id, name, role, color }` — email is explicitly excluded. Names and colors are already visible in assignment `assignedTo` includes. No PII exposure. |
</threat_model>
