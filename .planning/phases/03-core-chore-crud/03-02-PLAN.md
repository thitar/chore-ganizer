    Implement the template CRUD service and routes. The service owns all business logic; routes are thin HTTP wrappers (RESEARCH Pattern 1 + 2). Write tests FIRST (RED), then implement (GREEN), then refactor (REFACTOR).

    **Service: create(data)**
    - Input: `{ title: string, description?: string, points: number, category: string, createdById: number }`
    - Behavior: Creates a ChoreTemplate via `prisma.choreTemplate.create({ data })`. Returns the created template.
    - Test: Call `create(validData)` → returns template with auto-generated id and createdAt.
    - Test: (Route integration) POST /api/templates with valid body → 201, `{ success: true, data: { id, title, ... } }`.
    - Test: (Route integration) POST /api/templates without auth → 401.
    - Test: (Route integration) POST /api/templates with CHILD role → 403 (D-18).

    **Service: getAll()**
    - Input: None.
    - Behavior: Returns all templates via `prisma.choreTemplate.findMany({ orderBy: { title: 'asc' } })`. Any authenticated user can list templates (D-03: no creator-only restriction, D-18: GET is open to all authenticated).
    - Test: `getAll()` returns array of templates sorted by title.
    - Test: (Route integration) GET /api/templates → 200, array of templates.
    - Test: (Route integration) GET /api/templates without auth → 401.

    **Service: update(id, data)**
    - Input: `id: number, data: { title?: string, description?: string, points?: number, category?: string }`
    - Behavior: Updates the template via `prisma.choreTemplate.update({ where: { id }, data })`. Throws `AppError('Template not found', 404)` if id doesn't exist.
    - Cascade (D-01): Template edits cascade to ALL pending assignments. Since ChoreAssignment stores NO title/category/points — these come from the template JOIN via `choreTemplateId` — the cascade is AUTOMATIC at query time. Completed assignments read from `pointsAwarded` snapshot (D-02) and are unaffected. NO explicit `updateMany` on ChoreAssignment is needed (RESEARCH Pitfall 3).
    - Test: (Unit) `update(1, { title: 'New Title' })` → returns updated template with new title. Mock Prisma.
    - Test: (Route integration) PUT /api/templates/:id with valid body → 200, updated template.
    - Test: (Route integration) PUT /api/templates/:id after completing an assignment from that template → completed assignment's `pointsAwarded` unchanged, pending assignments reflect new template data in response include.
    - Test: (Route integration) PUT /api/templates/99999 → 404.

    **Service: delete_(id)** (named `delete_` to avoid JS keyword conflict)
    - Input: `id: number`
    - Behavior: Checks if any COMPLETED assignments reference this template. If yes → throws `AppError('Cannot delete template with completed assignments. Uncomplete them first.', 409)` (D-15). If no completed assignments → uses `prisma.$transaction` to: (1) delete all pending assignments via `prisma.choreAssignment.deleteMany({ where: { choreTemplateId: id, status: 'PENDING' } })`, (2) delete the template via `prisma.choreTemplate.delete({ where: { id } })`. Returns `{ deleted: true }` (D-05: cascade-delete pending, D-14: hard delete).
    - Test: (Unit) `delete_(1)` with no completed assignments → `{ deleted: true }`, deletes template + pending assignments. Mock prisma.$transaction.
    - Test: (Unit) `delete_(1)` with completed assignments → throws AppError 409.
    - Test: (Route integration) DELETE /api/templates/:id with only pending assignments → 200, template gone from DB.
    - Test: (Route integration) DELETE /api/templates/:id with completed assignments → 409, `{ success: false, error: { message: 'Cannot delete template with completed assignments...' } }`.
    - Test: (Route integration) DELETE /api/templates/99999 → 404.

    **Routes pattern** (follow auth.routes.ts exactly — RESEARCH Pattern 1):
    - `router.post('/', authenticate, authorize('PARENT'), validate(createTemplateSchema), async (req, res, next) => { ... })`
    - `router.get('/', authenticate, async (req, res, next) => { ... })`
    - `router.put('/:id', authenticate, authorize('PARENT'), validate(updateTemplateSchema), async (req, res, next) => { ... })`
    - `router.delete('/:id', authenticate, authorize('PARENT'), async (req, res, next) => { ... })`
    - Parse `req.params.id` as integer via `parseInt(req.params.id, 10)`.
    - All handlers use `try { ... } catch (err) { next(err) }` pattern.
    - Success responses use the standard envelope: `res.json({ success: true, data: ..., error: null })` or `res.status(201).json(...)` for create.

    **Test patterns** (follow auth.test.ts — RESEARCH Pattern 7):
    - Unit tests in `__tests__/services/template.service.test.ts`: mock Prisma via `jest.mock('../../config/prisma', () => ({ prisma: { choreTemplate: { ... }, choreAssignment: { ... } } }))`. Import mocked `prisma` via `require()`. Test each service function in isolation.
    - Integration tests in `__tests__/templates.test.ts`: use `request(app)` from supertest. Login as parent/child to get session cookie. Test full request → response cycle with live dev.db. Reset test data between suites.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API (POST/PUT/DELETE /api/templates) | Only PARENT role may cross this boundary; Zod validates all request bodies |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-01 | Elevation of Privilege | POST/PUT/DELETE /api/templates | mitigate | `authorize('PARENT')` middleware on all mutating routes; CHILD gets 403 (D-18) |
| T-3-02 | Tampering | Template delete bypass via race condition | mitigate | `prisma.$transaction` wraps existence check + cascade delete atomically; single-threaded Node.js event loop eliminates practical race window |
| T-3-03 | Tampering | Mass assignment via req.body on create/update | mitigate | Zod `createTemplateSchema`/`updateTemplateSchema` define exact allowed fields; `validate()` middleware strips unknown keys before service receives data |
| T-3-04 | Information Disclosure | Template listing reveals another user's templates | accept | D-03 explicitly allows any parent to view all templates; no sensitivity in template names/categories |
</threat_model>
