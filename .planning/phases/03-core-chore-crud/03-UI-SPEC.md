## Spacing Scale

Documented from existing codebase patterns (Phase 1–2). All values are multiples of 4. The established `px-3` (12px) on inputs and buttons is the sole exception — preserved for consistency with shipped Phase 1–2 code.

| Value | Tailwind Class | Where Used |
|-------|---------------|------------|
| 4px | `gap-1`, `mb-1` | Icon-to-text gap, label bottom margin |
| 8px | `py-2`, `gap-2`, `mb-2`, `mt-2` | Button/input vertical padding, compact gaps |
| 12px | `px-3` | Button/input horizontal padding (PHASE 1–2 LEGACY — do not change) |
| 16px | `px-4`, `p-4`, `gap-4`, `mb-4`, `mt-4` | Nav padding, card content, section gaps |
| 24px | `p-6` | Compact card padding, modals |
| 32px | `p-8`, `py-8` | Card padding (login card), page content vertical |
| 48px | `h-12 w-12` | Large icons (logo) |
| 64px | (unused) | Reserved for future page-level spacing |

**Rule:** New components use `px-3 py-2` for inputs and small buttons (matching LoginPage). New cards use `p-6` (tighter than login `p-8` for data-dense content) or `p-4` for compact cards.

## Typography

No custom CSS — all sizes via Tailwind utility classes. Exactly 3 text sizes, 2 font weights.

| Role | Tailwind Class | Size | Weight | Line Height | Usage |
|------|---------------|------|--------|-------------|-------|
| Body | `text-base` | 16px | 400 (normal) | 1.5 | Page descriptions, table cell content, form helper text |
| Heading | `text-2xl font-bold` | 24px | 700 (bold) | 1.2 | Page titles (`<h2>`), section headings, nav bar app name |
| Small | `text-sm` | 14px | 400 (normal) | 1.25 | Form labels, filter labels, table headers, nav links, card titles |

**Additional (derived, not independent roles):**
- **Caption/Timestamp:** `text-xs` (12px / 400) — status badges, timestamp labels. Same weight as Body/Small.
- **Nav Title (bold):** `text-2xl font-bold` — 24px / 700, same as Heading. Used in nav bar for app name.

**Rules:**
- 2 font weights only: `font-normal` (400) for body text and labels; `font-bold` (700) for headings and emphasis
- 3 primary font sizes: 24px (heading), 16px (body), 14px (small). 12px for captions/badges only.
- No `font-medium` (500) or `font-semibold` (600) — use `font-normal` or `font-bold`
- Body text always `text-gray-600` on light backgrounds
- Headings always `text-gray-900`

### Primary Palette (Indigo)

| Token | Hex | Tailwind Class | Usage |
|-------|-----|---------------|-------|
| `primary` | `#4F46E5` | `bg-primary`, `text-primary` | Primary buttons, active nav links, logo icon |
| `primary-hover` | `#4338CA` | `hover:bg-primary-hover` | Button hover states |
| `primary-light` | `#EEF2FF` | `bg-primary-light` | Selected row backgrounds, active filter chips |
| `primary-ring` | `#6366F1` | `focus:ring-primary-ring` | Focus rings on inputs, buttons, selects |

### Surface Palette

| Role | Hex | Tailwind Class | Usage |
|------|-----|---------------|-------|
| **Dominant (60%)** | `#F9FAFB` | `bg-gray-50` | All page backgrounds |
| **Secondary (30%)** | `#FFFFFF` | `bg-white` | Cards, nav bar, form surfaces, modals, dropdowns |
| **Accent (10%)** | `#4F46E5` | `bg-primary` / `text-primary` / `border-primary` | See "Accent reserved for" below |

### Semantic Colors

| Role | Hex | Tailwind Class | Usage |
|------|-----|---------------|-------|
| **Destructive** | `#DC2626` | `text-red-600`, `bg-red-600`, `hover:text-red-600` | Delete buttons, error text, destructive action hover |
| **Destructive light** | `#FEF2F2` | `bg-red-50` | Error message backgrounds |
| **Success** | `#16A34A` | `text-green-600` | Completion confirmation, "Completed" status badges |
| **Success light** | `#F0FDF4` | `bg-green-50` | Completed row backgrounds, success message backgrounds |
| **Warning/Muted** | `#9CA3AF` | `text-gray-400` | Disabled state, placeholder text |
| **Border** | `#D1D5DB` | `border` (default) | Input borders, card borders, table dividers |

### Accent Reserved For

The `primary` #4F46E5 color may ONLY appear on these elements:

1. Primary action buttons (`bg-primary text-white`)
2. Focus rings (`focus:ring-2 focus:ring-primary-ring`)
3. Logo icon on login page
4. Active navigation link indicator (`text-primary` or `border-b-2 border-primary`)
5. Loading spinner border (`border-primary`)
6. Selected/pressed filter chip or tab

Accent must NOT be used for: regular text, icons (except logo), decorative elements, backgrounds (except `primary-light` for selection state), or non-primary buttons.

**Contrast check:** `#4F46E5` on `#FFFFFF` = 6.0:1 ✅. `#4338CA` on `#FFFFFF` = 6.5:1 ✅.

## Interaction Contracts

### Visual Hierarchy

Each page follows a z-pattern reading flow:
1. **Nav bar** (top) — primary navigation, fixed identity
2. **Page title** — immediately below nav, establishes context
3. **Primary action button** (parent pages) — the visual anchor; the eye should land here second after the title
4. **Filter bar** (assignment/my-chores pages) — filter controls before content
5. **Content area** — tables/cards/list, the bulk of the page

**Focal points by page:**
- **Templates:** Empty state heading ("No chore templates yet") → "Create Template" button is the primary focal point once the user understands the empty state
- **Assignments:** For empty state, "Assign Chore" button. For populated state, the filter bar's user/status selects are the primary interaction point
- **My Chores:** "Mark Complete" buttons on pending chores are the primary focal point — the main action the user is here to take
- **Dashboard:** The "Upcoming Chores" section heading draws the eye to the user's immediate tasks

### Form Submission Flow

1. User fills form → clicks Save Template / Save Assignment
2. Button disables with "Saving..." text, `disabled:opacity-50`
3. API call fires
4. **Success:** form closes or collapses; brief green success message appears; list refreshes
5. **Error:** form stays open; red error panel appears above form with specific message
6. **Network timeout:** generic error panel + "Check your connection and try again."

### Delete Flow

1. User clicks delete icon (Trash2, `text-gray-400 hover:text-red-600`)
2. Inline red confirmation panel appears **below the item being deleted**
3. Panel has two buttons: "Delete" (red) and "Keep Template" / "Keep Assignment" (gray outlined)
4. **Cancel:** panel disappears, no change
5. **Delete:** button shows "Deleting..." `disabled:opacity-50`; API call fires
6. **Success:** row disappears with brief green message
7. **Error:** red message in the confirmation panel instead of removing it

### Complete Flow (My Chores)

1. User clicks "Mark Complete" on a pending chore
2. Button changes to "Completing..." with spinner icon, disabled
3. API call fires
4. **Success:** row updates — status badge becomes green "Completed", button disappears, green success flash
5. **Error:** button re-enables, red error text appears: "Could not complete chore. Try again."
6. **Idempotent:** clicking "Mark Complete" on an already-completed chore shows "Already completed" (should not be possible via UI since button is hidden, but guard in API layer)

### Filter Interaction

1. User changes a filter value → filters are applied live (no submit button)
2. If no results match filters → empty state: "No assignments match your filters." + "Clear filters" link
3. "Clear filters" resets all selects to "All" and clears date range inputs
4. Filters persist when navigating away and back (optional — React state only, no URL params needed for this phase)

### Navigation

- Nav bar links appear on all authenticated pages
- Active page's link is visually distinct (primary color + bottom border)
- Child users see only: Dashboard, My Chores
- Parent users see: Dashboard, My Chores, Templates, Assignments
- Clicking the "Chore-Ganizer" logo text navigates to `/`

## Responsive Behavior

### Breakpoint Strategy

One breakpoint: `sm` (640px). Below 640px is "mobile," above is "desktop."

| Element | Desktop (≥640px) | Mobile (<640px) |
|---------|-------------------|-----------------|
| Page padding | `px-4 sm:px-6 lg:px-8` | `px-4` |
| Nav bar | Horizontal links row | Links wrap or use hamburger menu (defer to Phase 7; horizontal row with `flex-wrap` is acceptable for Phase 3) |
| Filter bar | Single row, `flex-wrap` | Stacked, full-width selects |
| Table/list grid | Multi-column grid | Stacked cards per item instead of rows |
| Form cards | `max-w-lg` centered | Full width |
| Buttons | Inline `gap-2` | Full width `w-full`, stacked |

### Mobile Card Layout (Alternative to Table)

When viewport <640px, assignment/template lists switch from grid rows to stacked cards:

```html
<!-- Mobile card (replaces table row at <640px) -->
<div class="bg-white border rounded-lg p-4 space-y-2">
  <div class="flex justify-between">
    <span class="font-bold text-gray-900">{chore name}</span>
    <span class="text-sm">{status badge}</span>
  </div>
  <div class="text-sm text-gray-500">{due date}</div>
  <div class="flex justify-between items-center">
    <span class="text-sm">{points} pts</span>
    <button class="bg-primary text-white text-sm px-3 py-1 rounded-lg">Mark Complete</button>
```

## What NOT to Build in Phase 3

| Out of Scope | Reason |
|-------------|--------|
| Recurring chores | Phase 4 |
| Points display/balance | Phase 5 (though completing a chore writes points to DB, the UI doesn't show balance yet) |
| Calendar view | Phase 5 |
| User management pages | Phase 6 |
| Mobile hamburger menu | Phase 7 (flex-wrap nav is acceptable for Phase 3) |
| Loading skeletons | Phase 7 (spinner-only loading is acceptable for Phase 3) |
| Form validation beyond HTML `required` | Phase 7 (backend Zod schemas enforce validation; frontend just marks fields required) |
| Toast/notification system | Never — brief inline success messages only |
| Pagination | Never — family-scale data (fewer than 100 rows) means all data fits one page |
| Drag-and-drop reordering | Never — out of scope |
