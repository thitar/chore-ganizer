    Modify backend/src/middleware/errorHandler.ts to ensure production error responses never leak stack traces, internal error messages, or implementation details.

    Current behavior: The errorHandler already partially sanitizes the final 500 fallback (`process.env.NODE_ENV === 'production' ? 'Internal server error' : err.message`), but the AppError handler and Prisma error handler always return the raw error message.

    Changes needed:
    1. In the AppError handler (lines 37-46), wrap the response to sanitize in production:
       ```typescript
       if (err instanceof AppError) {
         const isProduction = process.env.NODE_ENV === 'production'
         res.status(err.statusCode).json({
           success: false,
           error: {
             message: isProduction && err.statusCode >= 500 ? 'Internal server error' : err.message,
             code: err.code,
           },
         })
         return
       }
       ```
       Rule: In production, only 5xx AppErrors show "Internal server error". 4xx AppErrors (like validation errors) can still show the message since they are user-facing.

    2. In the Prisma error handler (lines 49-71), the messages are already safe ("Resource already exists", "Resource not found"), so no changes needed.

    3. In the unknown error handler (lines 79-86), ensure the stack trace is never included:
       The existing code already does this correctly for the message, but verify the response object does NOT contain a `stack` field in ANY branch.
       Add an explicit safeguard: ensure no response path includes `stack: err.stack` or similar.

    4. Add a helper function at the top of the file (after imports) to keep the logic DRY:
       ```typescript
       const getSafeErrorMessage = (err: Error | AppError, statusCode: number): string => {
         if (process.env.NODE_ENV !== 'production') return err.message
         return statusCode >= 500 ? 'Internal server error' : err.message
       }
       ```
       Refactor all three error response sites to use this helper.

    Do NOT log sanitized messages — the logger should still receive the full error details (it already does). Only the HTTP response is sanitized.
  </action>
