# Phase 4 — UI Design Contract

> Visual and interaction contract for RecurringChoresPage. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Parent phase deliverable:** Parents create/delete recurring chores; existing pages (MyChores, Assignments, Dashboard) auto-display occurrences.
>
> **Inherits:** All patterns from `03-UI-SPEC.md` (design system, spacing, typography, color, component inventory, loading states, copy patterns). This spec adds ONLY the new page contract.

---

## Inheritance Notice

This phase reuses the design system established in Phase 3 without modification:

- **Design system:** Vanilla Tailwind CSS, lucide-react icons, Inter font
- **Spacing scale:** All values multiples of 4 (12px horizontal padding on inputs is the Phase 1–2 legacy exception)
- **Typography:** 3 sizes (24/16/14px) + caption (12px), 2 weights (400/700) — no `font-medium` or `font-semibold`
- **Color:** 60/30/10 ratio — gray-50 page bg, white cards, indigo primary reserved for actions/focus/active
- **Component patterns:** All Phase 3 patterns (Navigation Bar, Card, Form Input, Button, Status Badge, Table/List, Filter Bar, Confirmation Panel, Loading State)

**Rule:** New code MUST follow Phase 3 patterns exactly. No new component primitives invented. No new color tokens. No new spacing values.

## Type Discriminator Display (MyChoresPage, AssignmentsPage, Dashboard)

The `type: 'REGULAR' | 'RECURRING'` field is a backend implementation detail. **It MUST NOT be displayed to the user.** Pages render occurrences identically to regular assignments — the user sees the chore title, due date, status badge, and complete button. The discriminator only routes the completion API call internally.

**No UI changes required** to MyChoresPage, AssignmentsPage, or DashboardPage for the type field. They use the same `useAssignments` hook and render the same `assignments` array.

**Optional visual hint (decide at implementation time):** A small "↻" icon could appear next to recurring occurrence titles. If implemented, it must:
- Use `text-gray-400` (muted)
- Be `h-3 w-3` (smaller than text)
- Have `aria-label="Recurring chore"`
- Be placed inline with the title using `inline-flex items-center gap-1`

If the icon adds complexity, skip it — UX is fine without the visual distinction.

## States Summary

| State | Visual |
|-------|--------|
| Loading (initial fetch) | Card body spinner, `animate-spin h-6 w-6 border-b-2 border-primary` |
| Loading (mutation) | Spinner inside button, button text replaced with "Creating..." or "Deleting..." |
| Empty | "No recurring chores yet. Create one above to get started." |
| Error | `bg-red-50 border border-red-200 rounded-lg p-4` with retry button |
| Form validation error | `bg-red-50 border border-red-200 rounded-lg p-3` below form |
| Submitting | Submit button disabled (`disabled:opacity-50`) |
| Delete confirm open | Inline `bg-red-50` panel below the row |
| Network error on delete | Error toast/banner (Phase 3 pattern: top-of-page red banner with message) |

## Accessibility

- All form inputs have `<label>` elements (no placeholder-only labels)
- All icon buttons have `aria-label`
- Form submit can be triggered with Enter (default browser behavior — no custom handling needed)
- Radio group: `<fieldset>` with `<legend>` (or visible label) for grouping
- Selects use native `<select>` (keyboard-navigable by default)
- Color contrast: All text on `bg-gray-50` is `text-gray-600` (4.5:1+) or `text-gray-900` (15:1+); primary buttons have white text on indigo (6.0:1)
- Focus visible: `focus:ring-2 focus:ring-primary-ring` on all interactive elements

## Anti-Patterns (Don't Do)

- ❌ Don't add a separate "Recurring" page for child to view — they see occurrences in MyChoresPage already
- ❌ Don't show the `type` discriminator to the user — it's backend-only
- ❌ Don't use a modal overlay for delete confirmation — Phase 3 established inline panel pattern
- ❌ Don't add a calendar widget for day-of-month — use native `<input type="number">`
- ❌ Don't add a new color for "recurring" — use existing palette
- ❌ Don't use a table element (`<table>`) — Phase 3 uses grid-based card pattern
- ❌ Don't add bulk operations (create many at once) — out of RECUR scope
- ❌ Don't add recurring chore editing — current implementation is create+delete; defer edit to polish phase
- ❌ Don't paginate — family scale (<10 recurring chores) doesn't need pagination
