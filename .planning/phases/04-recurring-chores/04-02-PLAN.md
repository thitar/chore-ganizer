  <behavior>
    Write tests FIRST (RED), then implement (GREEN).

    **Service: create(data)**
    - Input: `{ choreTemplateId: number, assignedToId: number, frequency: 'DAILY' | 'WEEKLY' | 'MONTHLY', dayOfWeek?: number, dayOfMonth?: number }`
    - Validates template exists (throw 404 if not). Creates RecurringChore. Returns created chore with template include.
    - Test: (Route) POST /api/recurring as PARENT → 201 with chore data including template.
    - Test: (Route) POST /api/recurring as CHILD → 403.
    - Test: (Route) POST /api/recurring with non-existent templateId → 404.

    **Service: generateOccurrences(from, to)**
    - Input: `from: Date, to: Date` — the date range to generate for.
    - Algorithm per RESEARCH: Find all RecurringChore records. For each, compute which dates in [from, to] should have an occurrence based on frequency. Query existing occurrences for those dates via `@@unique`. Create missing ones with status PENDING and assignedToId from the chore.
    - Daily: every date in range.
    - Weekly: every day matching `dayOfWeek` in range.
    - Monthly: every day matching `dayOfMonth` in range (clamped to month end if dayOfMonth > days in month).
    - Test: (Unit) Daily chore generates 30 occurrences for 30-day range (minus existing).
    - Test: (Unit) Weekly chore (Monday) generates 4 occurrences for a 30-day range with 4 Mondays.
    - Test: (Unit) Monthly chore (day 15) generates 1 occurrence for a single-month range.
    - Test: (Unit) Idempotent — calling twice produces no duplicates (mock existing occurrences).

    **Service: completeOccurrence(occurrenceId, userId)**
    - Input: `occurrenceId: number, userId: number`
    - Owner check: `occurrence.assignedToId !== userId` → 403.
    - Already-completed check: `occurrence.status === 'COMPLETED'` → 409.
    - Transactional (callback $transaction): update status to COMPLETED, set pointsAwarded = template.points, create EARNED PointLog.
    - Test: (Unit) Owner completes → status COMPLETED, pointsAwarded set, PointLog EARNED in transaction.
    - Test: (Unit) Non-owner → throws 403.
    - Test: (Route) POST /api/recurring/:id/complete as owner → 200.
    - Test: (Route) POST /api/recurring/:id/complete as non-owner child → 403.

    **Service: delete_(id)**
    - Input: `id: number`
    - Delete PENDING occurrences first: `prisma.recurringOccurrence.deleteMany({ where: { recurringChoreId: id, status: 'PENDING' } })`
    - COMPLETED occurrences survive (no onDelete Cascade — they stay with `recurringChoreId` set to the now-deleted chore? No — they have the FK. Need to handle: either setNull or accept that the FK constraint blocks deletion).
    - **Design decision:** Remove `onDelete: Cascade` already done in 04-01. The FK from RecurringOccurrence to RecurringChore must allow `recurringChoreId` to become unusable. Use `onDelete: SetNull` on the RecurringOccurrence→RecurringChore relation (change in schema). This way completed occurrences lose the FK reference but data is preserved.
    - After that: `prisma.recurringChore.delete({ where: { id } })`. Return `{ deleted: true }`.
    - Test: (Unit) Delete with no completed occurrences → { deleted: true }, PENDING deleted, chore deleted.
    - Test: (Unit) Delete with completed occurrences → PENDING deleted, completed occurrences survive with null recurringChoreId, chore deleted.
    - Test: (Route) DELETE /api/recurring/:id → 200.

    **GET /api/recurring**
    - Returns all recurring chores with `include: { template: { select: { id, title, points, category } }, assignedTo: { select: { id, name, color } } }`.
    - Test: (Route) GET /api/recurring → 200, array.
